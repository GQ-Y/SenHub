# 系统架构优化方案

## 一、问题分析

### 1. SDK库文件目录不规范
- **问题**：`lib/`目录下已有SDK库文件，但`MakeAll/`目录也存在重复的SDK文件
- **影响**：目录混乱，维护困难，可能导致库文件加载错误
- **解决方案**：合并`MakeAll/`到`lib/hikvision/`，统一管理所有SDK库文件

### 2. 文件存储路径分散
- **问题**：抓图、录制、下载文件分别存储在`captures/`、`records/`、`downloads/`目录
- **影响**：缺乏统一管理，配置分散，不利于备份和清理
- **解决方案**：统一到`storage/`目录下，按类型分类：
  - `storage/captures/` - 抓图文件
  - `storage/records/` - 录制文件
  - `storage/downloads/` - 下载文件

### 3. 数据库设备管理兼容性问题
- **问题**：不同品牌设备字段规范不统一，可能导致数据不一致
- **影响**：设备管理可能出现异常，品牌识别错误
- **解决方案**：
  - 统一品牌字段值：`hikvision`、`tiandy`、`dahua`
  - 添加数据验证和迁移脚本
  - 统一设备ID生成规则

### 4. Docker配置需要优化
- **问题**：Docker配置未适配新的目录结构
- **影响**：容器内路径映射错误，可能导致文件存储失败
- **解决方案**：更新所有Docker相关文件，适配新的存储路径

### 5. 多余文件清理
- **问题**：存在重复的测试文件（如`CaptureTest.java`）
- **影响**：代码冗余，维护困难
- **解决方案**：删除重复文件，保留必要的测试代码

## 二、新需求实现

### 1. 报警信号自动抓图
- **需求**：根据海康、天地伟业SDK的报警信号输入，自动触发抓图
- **实现方案**：
  - 实现报警回调处理器
  - 集成到各品牌SDK的报警回调中
  - 自动调用抓图服务

### 2. OSS上传功能
- **需求**：抓图后自动上传到OSS
- **实现方案**：
  - 创建OSS上传服务
  - 集成到抓图流程中
  - 支持配置开关和重试机制

## 四、目录结构优化

### 优化前
```
server/
├── lib/
│   ├── hikvision/
│   ├── tiandy/
│   └── dahua/
├── MakeAll/          # 重复的SDK文件
├── captures/         # 抓图文件
├── records/          # 录制文件
└── downloads/        # 下载文件
```

### 优化后
```
server/
├── lib/
│   ├── hikvision/    # 包含MakeAll中的文件
│   ├── tiandy/
│   └── dahua/
├── storage/          # 统一存储目录
│   ├── captures/     # 抓图文件
│   ├── records/      # 录制文件
│   └── downloads/    # 下载文件
└── data/             # 数据库文件
```
